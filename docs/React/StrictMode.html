StrictMode

參考資料：

React Document: StrictMode

React 18 useEffect Double Call for APIs: Emergency Fix

前言​

在開發時，有時候會發現 useEffect 會在第一次 render 時，被觸發兩次，上 stack overflow 找，大部分都會說把包在外層的 <React.StrictMode></React.StrictMode> 拿掉，但並不了解為什麼這樣就可以正常開發，這次就來瞭解這個 Components 的特性與使用時機。

特性​

  • React 18 之後才有的 future
  • 用途在於可以及早發現錯誤
  • 僅在開發模式下有用（Prod 不受影響）

檢查項目​

  • re-render component: 檢查 render 是否是 pure 的
  • re-run effect: 檢查 effect 裡面是否是 pure 的
  • checked usage api: 檢查是否有使用已經棄用的 React API

注意事項​

  • 沒有 props 的參數
  • 因為沒有 props 的參數，所以不能透過 props 動態決定是否要啟用 StrictMode ，所以當團隊開發完，可以決定是否要在開發環境移除 StrictMode 或是將 StrictMode 移動到需要的 Component 外層就好

用法​

包在整個 APP 外層
import {StrictMode} from "react";
import {createRoot} from "react-dom/client";

const root = createRoot(document.getElementById("root"));
root.render(
  <StrictMode>
    <App />
  </StrictMode>
);
包在各別 Component 外面
import {StrictMode} from "react";

export default function Layout() {
  return (
    <main>
      <StrictMode>
        <Button />
      </StrictMode>
      <StrictMode>
        <Navbar />
      </StrictMode>
    </main>
  );
}

上面的範例看起來沒什麼問題，在 App 初始化的時候去 connection 連線，但是當今天的需求出現，我們可能會動態改變 room Id 的話，上面的寫法就會出現問題了。

我們加入切換 roomId 的 select，當 select 變動時，會重新去連線。

const serverUrl = "https://localhost:1234";


function ChatRoom({roomId}) {
  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.connect();
  }, [roomId]);


  return <h1>Welcome to the {roomId} room!</h1>;
}


let connections = 0;


function App() {
  const [roomId, setRoomId] = useState("general");
  const [show, setShow] = useState(false);
  return (
    <>
      <label>
        Choose the chat room:{" "}
        <select value={roomId} onChange={(e) => setRoomId(e.target.value)}>
          <option value="general">general</option>
          <option value="travel">travel</option>
          <option value="music">music</option>
        </select>
      </label>
      <button onClick={() => setShow(!show)}>
        {show ? "Close chat" : "Open chat"}
      </button>
      {show && <hr />}
      {show && <ChatRoom roomId={roomId} />}

}

會發現，當切換時，不斷跳 connect 的 console ，但是沒有 disconnect ，代表每次的連線都沒有把舊的連線清掉，並且一直新增連線，這會造成後端的負擔太大，因為會有許多沒在用的連線一直在背景。

當我們一樣把 StrictMode 加在 App 外面時，會發現在第一次進來時，就會看到 connect 被觸發兩次，代表他 re-run 了 useEffect ，去檢查 effect 裡面的 callback 是否是 pure 的。

要解決以上的方式就是，要在 useEffect 的 return 去執行 disconnect ，當連線改變時，會先斷掉上一個連線才會進行新的連線。

Solution
useEffect(() => {
  const connection = createConnection(serverUrl, roomId);
  connection.connect();
  return () => connection.disconnect();
}, [roomId]);

解決 useEffect 會 call 兩次 API 的問題​

回到最源頭，StrictMode 有以上特性，那就要解決我在開發時，在 useEffect call 兩次 API 的問題

這邊提供一些我參考並實作，可以解決此問題的方式

Remove StrictMode​

最快的方式就是，移除 StrictMode 的 component ，因為移除也不影響開發，主要是在 debug 時會比較需要測試較多案例

移除 StrictMode
import {createRoot} from "react-dom/client";

const root = createRoot(document.getElementById("root"));
root.render(<App />);

Use AbortController​

使用 AbortController ，這是一個實驗中的功能，可以在需要的時候中斷請求，以下面的案例中，我們在 useEffect 的 return 去中止這個 API ，避免在第二次 re-run 時，再發一次 API

AbortController
import React, {useState, useEffect} from "react";

const Demo = () => {
  const [response, setResponse] = useState({});

  useEffect(() => {
    const controller = new AbortController();
    fetch("your api end point", {
      signal: controller.signal,
    })
      .then((res) => res.json())
      .then(setResponse);
    return () => controller.abort();
  }, []);

  return <div>test api post</div>;
};

export default Demo;

可以看到，透過 AbortController 可以避免第二次 API 發出去。

Use React-Query​

可以使用 react-query 這個 library 處理 api 的 get/post，除了可以避免打兩次 API 外，也提供許多 api 狀態的參數供頁面使用。

react-query demo
import React from "react";
import {useQuery} from "react-query";
const Demo = () => {
  const {isLoading, error, data} = useQuery("api", () =>
    fetch("your api end point").then((res) => res.json())
  );

  console.log(data);

  return <div>1234</div>;
};

export default Demo;
